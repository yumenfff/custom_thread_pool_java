# Курсовая работа (Часть 1): Кастомный Thread Pool на Java

## Что реализовано

Собственный пул потоков с настраиваемыми параметрами,
балансировкой задач, логированием и политикой обработки перегрузки

---

## Параметры пула

| Параметр | Описание                                       |
|---|------------------------------------------------|
| `corePoolSize` | Базовое количество потоков                     |
| `maxPoolSize` | Максимальное количество потоков                |
| `keepAliveTime` | Время простоя лишнего потока перед завершением |
| `timeUnit` | Единицы измерения времени                      |
| `queueSize` | Максимальный размер очереди каждого воркера    |
| `minSpareThreads` | Минимальное число свободных потоков в запасе   |

---

## Как работает пул

1. При создании сразу запускается `corePoolSize` потоков
2. При вызове `execute()`:
    - если `minSpareThreads` не соблюдён, то создаётся новый поток (если не достигнут `maxPoolSize`)
    - задача кладётся в очередь воркера по Round Robin
    - если все очереди полны и потоков достигнут `maxPoolSize`, то вызывается политика отказа
3. Каждый Worker берёт задачи из своей личной очереди
4. Перед выполнением задачи Worker проверяет, что пул не завершается
5. Если Worker не получил задачу за `keepAliveTime` и потоков больше `corePoolSize`, то он завершается
6. При `shutdown()` новые задачи не принимаются, а уже добавленные выполняются до конца

## Балансировка задач

Реализована стратегия **Round Robin**: задачи раздаются воркерам по кругу через счётчик `AtomicInteger`.
У каждого воркера своя `BlockingQueue`, это позволяет равномерно распределять нагрузку

Если очередь выбранного воркера полна, то перебираем остальных по порядку

### Как работает распределение задач:

1. Поступает новая задача
2. Счётчик rrCounter увеличивается: 0 -> 1 -> 2 -> 0 -> 1 ...
3. Вычисляется индекс воркера: `index = rrCounter % количество воркеров`
4. Задача кладётся в очередь выбранного воркера
5. Если очередь полна -> перебор остальных воркеров по порядку
6. Если все очереди полны -> вызывается политика отказа

## Обработка перегрузки

Реализованы две политики через интерфейс `RejectedTaskHandler`:

- **CallerRunsPolicy** используется по умолчанию в пуле: задача запускается прямо в потоке вызывающего.
Задача не теряется, но вызывающий поток блокируется, это снижает скорость отправки новых задач
и естественным образом замедляет нагрузку на пул


- **AbortPolicy** - альтернативная политика: бросает `RejectedExecutionException`.
Используется в демонстрации перегрузки, чтобы явно показать отказы и посчитать их количество

Логируются все ключевые события:

```text

[Worker] MyPool-worker-1 executes DemoTask-3

[Worker] MyPool-worker-1 terminated
[Rejected] Task DemoTask-7 was rejected due to overload!
```

## Интерфейс управления

```text
pool.execute(runnable);  - запуск без результата
pool.submit(callable);  - запуск с результатом (Future)
pool.shutdown();  - мягкое завершение
pool.shutdownNow();  - прерывание всех потоков
pool.awaitTermination(15, SECONDS);  - ожидание завершения
```

## Анализ производительности

#### Тест: 2000 CPU-задач, 4 потока

Логи в `Benchmark` отключены (флаг `enableLogs=false`), чтобы вывод в консоль не влиял на замер.
Для чистоты замера в бенчмарке используется маленький `keepAliveTime` (`1 ms`),
чтобы idle timeout не добавлял лишнюю задержку

```text
RESULTS:
CustomThreadPool : 14 ms
JDK ThreadPool : 7 ms
Difference : +7 ms (100.0%)
-> JDK пул быстрее
```

В этом тесте мой пул медленнее JDK-пула примерно в 2 раза.

Причины: отдельная очередь на каждого воркера, частая синхронизация при выборе очереди
и более простая реализация без низкоуровневых оптимизаций `ThreadPoolExecutor`

### Исследование параметров

Тестировалось на 1000 задачах с лёгкой CPU-нагрузкой

Методика: `1` прогрев + `5` измерений, в таблице указано среднее время (`avgTimeMs`)

| corePoolSize | maxPoolSize | queueSize | avgTimeMs |
|---|---|---|---|
| 2 | 4 | 10 | 4 |
| 4 | 4 | 100 | 3 |
| 4 | 8 | 50 | 2 |
| 8 | 8 | 200 | 5 |

Выводы:

- на этом запуске лучший результат дал вариант `core=4`, `max=8`, `queue=50`
- слишком большой пул (`8/8`) на лёгкой CPU-нагрузке дал хуже результат из-за накладных расходов
- слишком маленькая очередь (`10`) ухудшает время из-за частого переключения и fallback-логики
- практично начинать с `core` около числа ядер и дальше подбирать `max/queue` под тип нагрузки
